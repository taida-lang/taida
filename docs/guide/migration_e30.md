# E30 移行ガイド (D29 → `@e.30`)

> **対象**: D29 stable (`@d.29`) で書かれた Taida コードを E30 (`@e.30`) に移行する開発者
> **方針**: gen-E は **deprecation なし、即破壊的変更**。旧構文は E30 で reject されます。

---

## 概要

E30 は Taida の **gen-E 初の破壊的変更 stable** です。以下の主軸で言語仕様 (`.td` source code) が breaking change を受けます。

| 主軸 | 内容 | 関連 blocker |
|------|------|-------------|
| 1 | 型構文の surface 完全統一 (`Name[?type-args] [=> Parent] = @(...)` 単一構文) | E30B-001 |
| 2 | declare-only 関数フィールドの全系統許可 | E30B-002 |
| 3 | declare-only 関数フィールド未指定の compile-time 検出 (`[E1410]`) | E30B-003 |
| 4 | 関数のデフォルト値 (defaultFn) 定義 + 全関数型への自動生成 | E30B-004 |
| 5 | Docs 全面再構成 (3 系統 → class-like 単一概念) | E30B-005 |
| 6 | graph / pkg facade / LSP / introspection consumer parity | E30B-006 |
| 7 | addon sentinel の facade 明示 binding 化 (`RustAddon[...]`) | E30B-007 |

詳細仕様は新章 [04_class_like.md](04_class_like.md) および `.dev/E30_DESIGN.md` (公開 docs 化は別途) を参照してください。

---

## 移行手順

`@e.X` 以降の CLI は移行コマンドを提供しません。`.td` ソースコードの
移行は、このガイドの Before / After とチェックリストに従って手動で行います。

```bash
taida way check <PATH>
# Run your project-specific tests after the Taida gate passes.
```

古い RC 計画や作業メモに残る「移行コマンド」案は、`@e.X` の確定 CLI には含まれません。

## 主軸 1: 型構文の surface 統一

### Before (D29)

```taida
// 3 系統独立した surface 構文
Pilot = @(name: Str, age: Int)                     // TypeDef
Mold[T] => Result[T] = @(throw: Error)             // Mold 継承
Error => NotFound = @(msg: Str)                    // Error 継承
```

### After (E30)

```taida
// すべて class-like 単一構文
Pilot = @(name: Str, age: Int)                     // zero-arity sugar として保持
Pilot[] = @(name: Str, age: Int)                   // 等価形
Mold[T] => Result[T] = @(throw: Error)             // 継承構文として保持
Error => NotFound = @(msg: Str)                    // 継承構文として保持
```

### 何が変わるのか

**surface はほぼ変わりません**。同じコードがそのままパースされます。
変わるのは **概念モデル** です:

- 旧: TypeDef / Mold 継承 / Error 継承の **3 系統が独立** に存在
- 新: すべて **class-like 単一概念** の異なる形 (zero-arity / 親型あり継承)

実装側 (parser / checker / graph / lowering / consumer) は新概念モデルに統合されます (E30B-001/006)。

### Lock-B verdict による細部

#### Sub-B1: zero-arity sugar 許容

`Pilot = @(...)` は `Pilot[] = @(...)` の sugar。型引数を持たない場合は `[]` を省略できます。

#### Sub-B2: `Error =>` prefix の特別扱い撤廃

旧: `Error =>` は special syntax として parse されていた。
新: 通常の class-like 継承 (`Parent => Child = @(...)`) として解釈。Error は標準ライブラリの base 型のひとつ。

#### Sub-B3: 親型 arity 一致 + 子側型引数追加

```taida
// 親: 2 引数
Result[T, P] = @(...)

// OK: 親へ 2 引数で適用、子側で V 追加
Result[T, P] => CustomResult[T, P, V] = @(meta: V)

// NG: arity mismatch → [E1407]
Result[T] => Bad[T] = @(...)
Result[T, P, V] => Bad[T, P, V] = @(...)
```

## 主軸 2: declare-only 関数フィールドの全系統許可

### Before (D29)

```taida
// class-like (TypeDef): declare-only OK
Greeter = @(
  name: Str
  greet: Str => :Str   // OK
)

// Mold 継承: declare-only NG (本体必須)
Mold[T] => Transformer[T] = @(
  apply: T => :T   // ERROR: 本体必須
)

// Error 継承: declare-only NG (本体必須)
Error => Recoverable = @(
  msg: Str
  recover: Unit => :Unit   // ERROR: 本体必須
)
```

### After (E30)

```taida
// すべての系統で declare-only 許可
Greeter = @(name: Str, greet: Str => :Str)
Mold[T] => Transformer[T] = @(apply: T => :T)            // OK
Error => Recoverable = @(msg: Str, recover: Unit => :Unit)  // OK
```

declare-only 関数フィールドの default 値は、E30 で導入される **defaultFn 自動生成** で充足されます。

## 主軸 3: declare-only 関数フィールドの compile-time 検出 (`[E1410]`)

### Before (D29)

```taida
Pilot = @(name: Str, age: Int, greet: Str => :Str)

rei <= Pilot(name <= "Rei", age <= 14)
rei.greet("anyone")   // Runtime error: Unknown method 'greet'
```

D29 では default として空 pack `@()` が割り当てられ、呼び出し時に runtime error。

### After (E30)

```taida
rei <= Pilot(name <= "Rei", age <= 14)
rei.greet("anyone")   // "" (defaultFn が自動生成、戻り型 Str のデフォルト "")
```

defaultFn が **生成不可能な戻り型** (opaque / abstract external type) の場合は compile-time に reject されます。

```taida
// External / unknown 型は defaultFn 生成不能 → declare-only field では [E1410]
// (Opaque は import 経由の external 型で、TypeRegistry に登録されないケースを想定)
Foo = @(
  bad: Int => :Opaque   // [E1410]: declare-only function field requires
                         //         default function or explicit value
                         //         (return type cannot be auto-generated by defaultFn)
)
```

> `[E1410]` は E30 で「declare-only function field requires default function or explicit value」として発火します (E30 Phase 5 land 済)。旧意味 (InheritanceDef 子フィールドの再定義違反) は `[E1411]` に番号移動されました (E30 Phase 3 land 済)。発火条件: declare-only function field の戻り型が defaultFn 生成不能な opaque / unknown alias の場合、definition-site で発火します (instantiate 時ではない、定義時点で早期検出)。

## 主軸 4: defaultFn 自動生成

E30 では、すべての関数型に対して defaultFn が自動生成されます。

```taida
// 全関数型に defaultFn が attach される
// f: Int => :Float          → defaultFn: 0.0 を返す
// f: Str => :Str             → defaultFn: "" を返す
// f: Int Int => :Bool        → defaultFn: false を返す
// f: Int => :Lax[Int]        → defaultFn: Lax{value: 0, hasValue: false}
// f: Pilot => :NervStaff     → defaultFn: NervStaff のデフォルト pack
// f: Int => :(Int => :Int)   → defaultFn: 「Int => :Int の defaultFn」を返す
// f: Unit => :Unit           → defaultFn: no-op
// f: Int => :Async[Int]      → defaultFn: 即時 resolve (T のデフォルト値で)
```

詳細仕様は [09_functions.md](09_functions.md) の defaultFn 節を参照してください (Phase 6 実装と同期して追加)。

## 主軸 7: addon sentinel の facade 明示 binding

### Before (D29)

```taida
// terminal addon (taida-lang/terminal)
// terminalSize は暗黙の pre-inject (AST binding なし)
size <= terminalSize()
```

### After (E30)

```taida
// .td facade で明示 binding
///@ Purpose: ターミナルサイズを取得する
///@ Returns: @(cols: Int, rows: Int)
///@ Throws:
///@   - TerminalSizeNotATty: stdout が TTY でない場合
terminalSize <= RustAddon["terminalSize"](arity <= 0)

// 利用側は変わらない
size <= terminalSize()
```

`RustAddon[...]` 形式の束縛は addon 側の facade migration で明示的に追加します。E31 では AST migration tool は提供しません。

`native/addon.toml` の `[functions]` テーブルと `RustAddon[...]` 束縛の整合チェック（drift 検出）が `taida way check` / CI で必須になります（Lock-G）。

> **現行 note (E30 Phase 7 sub-track B、Sub-G4/B-5)**
>
> explicit binding surface は `RustAddon["fn"](arity <= N)` に一本化されています。
>
> - parser: `RustAddon["fn"](arity <= N)` が既存の `MoldInst` シェイプとして parse されるため、AST 拡張は不要 (parser probe `test_e30b_007_rust_addon_binding_*` で pin)
> - Interpreter: facade load context (`Interpreter::loading_addon_facade_ctx`) を導入し、`MoldInst("RustAddon", ...)` を addon sentinel `__taida_addon_call::<pkg>::<fn>` に解決 (`eval_rust_addon_binding`)
> - native / wasm-full codegen: `src/addon/facade.rs::load_facade_summary` が explicit binding を `local_aliases` に集約し、drift check + manifest fn 存在確認を summary 側で完結する
> - `[E1412]` は `RustAddon[...]` surface 自体の違反、`[E1413]` は legacy bare addon function reference の reject を表す
> - **legacy implicit pre-inject は撤廃済み**。manifest `[functions]` の関数名を bare 参照する facade は `[E1413]` で hard-fail するため、`name <= RustAddon["name"](arity <= N)` を facade に明示してください。
>
> 利用側 (`>>> taida-lang/terminal => @(...)`) の import 形は変わりません。変更が必要なのは addon facade 内部の binding です。

## 移行手順

### Step 1: D29 で動作確認

E30 移行前に、対象コードベースが D29 (`@d.29`) で動作することを確認してください。

```bash
# Taida CLI を D29 stable に固定
taida --version   # @d.29 を確認

cd <project>
taida way <PATH>     # green であること
# Run your project-specific tests here.
```

### Step 2: E30 RC を install

E30 RC が release されたら install します (`@e.30.rcM`)。

Taida 本体の導入・切り替え方法は配布形態に従ってください。E31 以降の
`taida upgrade` は self-upgrade 専用で、`--e.30.rcM` のような AST / version
移行フラグは提供しません。

### Step 3: 必要箇所を手動で書き換え

確認する項目は:

- 旧 `Mold[T] =>` / `Error =>` を含む既存構文を新統一構文に書き換え (実質的には surface ほぼ同じ、語彙の整理が主)
- 暗黙 addon pre-inject を `RustAddon[...]` binding に展開
- declare-only 関数フィールドが defaultFn 生成不可 (opaque 戻り型) の場合は `[E1410]` warning を出し、明示的な default 値の追加を促す

### Step 4: E30 で再検証

```bash
taida way <PATH>     # green であること
# Run your project-specific tests here.
```

### Step 5: stable 版に切り替え

`@e.30` (label なし、stable) が release されたら、配布形態に従って
Taida 本体を切り替えます。`taida upgrade` に version-selection flag はありません。

## breaking points 一覧 (E30 で reject される旧構文)

| 旧構文 | 旧 status | 新 E30 status | 移行先 |
|--------|----------|---------------|--------|
| `Mold[T] =>` を special parse | special syntax | 通常の継承構文 | parser 統合 (surface 不変) |
| `Error =>` を special parse | special syntax | 通常の継承構文 | parser 統合 (surface 不変) |
| Mold 継承で declare-only field | reject (本体必須) | OK | そのまま動く |
| Error 継承で declare-only field | reject (本体必須) | OK | そのまま動く |
| declare-only field を空 pack default | runtime error | compile-time `[E1410]` | defaultFn 自動生成、または明示 default |
| addon-backed function 暗黙 pre-inject | OK | reject | `RustAddon[...]` binding に書き換え |
| 親型適用 arity mismatch | (旧 [E1407] 別意味) | `[E1407]` arity mismatch | 親型適用引数を arity に合わせる |
| `Statement::TypeDef` / `MoldDef` / `InheritanceDef` 3 variant 別存在 (AST レベル) | 内部 3 variant | `ClassLikeDef` 単一 variant | AST 統合 (consumer は class-like 語彙へ移行、E30B-006) |
| `NodeKind::BuchiPackType` / `MoldType` / `ErrorType` (graph レベル) | 3 kind | `ClassLikeType` 単一 kind | graph 統合 (E30B-006) |

## よくある質問

### Q. surface 構文がほぼ同じなら、なぜ breaking change？

A. **概念モデルの統合** が breaking。AST / graph / consumer (LSP / introspection / docgen) は新 class-like 語彙に書き換えられます。古い AST kind を直接参照する外部ツール (ある場合) は更新が必要です。語彙不変な surface コード自体は基本的に手動確認で済み、CLI による AST 書き換えは提供しません。

### Q. なぜ deprecation period がない？

A. E gen は破壊的変更を許容する track 系列で、grace なし即破壊が方針 (Lock-E)。RC 期間 (`@e.30.rcM`) が事実上の grace です。

### Q. D29 stable で動くコードを E30 RC でテストするには？

A. 並行 install はサポートされません。一旦コピーして RC を install、検証完了後に stable へ切り替えるのが推奨ワークフローです。

### Q. addon の互換性は？

A. addon facade `.td` (`taida-lang/terminal` など) は `RustAddon[...]` binding に書き換える必要があります。Rust 側 (`addon.toml` + cdylib) は互換維持です (drift check で sync 確認)。
